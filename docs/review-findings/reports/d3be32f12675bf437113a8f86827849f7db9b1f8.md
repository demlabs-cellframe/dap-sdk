## Finding RF-2026-04-21-010
- Priority: P1
- Status: FIXED
- Location: `net/stream/stream/dap_stream.c:221-243` (callers at `:1241`, `:1427`, `:1446`; client stream construction path `:682-734`)

### What is wrong
`s_stream_esocket_is_detached()` was changed from a simple `_inheritor == NULL` check to:
- treat `_inheritor != NULL` as attached;
- otherwise require a stream with matching `esocket_uuid` to be present in global `s_streams`.

For client/datagram paths this is not a safe attachment predicate by itself:
- `dap_stream_new_es_client()` can create a live stream before it is guaranteed to be present in `s_streams`;
- when `_inheritor` is intentionally unset in these paths, helper can return `true` (detached) for still-live streams.

### Risk
Packet processing may exit early in normal flow (`goto cleanup` after channel callback/notifiers), causing dropped input packets and unstable handshake/data flow on UDP/datagram paths.

### Fix status
Fixed later in `eb9c56eabdea6254f497a7757adca93119ef2c4c` by adding explicit client FSM/trans_ctx stream check and UUID index-based lookup path.

### Suggested checks
- Re-run UDP client handshake/data scenarios with `_inheritor == NULL` transport context.
- Verify packet processing does not early-abort while stream remains reachable via client FSM/trans_ctx.
