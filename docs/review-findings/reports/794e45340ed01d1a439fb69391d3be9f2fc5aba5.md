## Finding RF-2026-04-21-008
- Priority: P2
- Status: FIXED
- Location: `tests/regression/net/trans/test_udp_multiclient_regression.c:98-113`, `tests/integration/net/trans/test_trans_helpers.h:66`

### What is wrong
Regression test now auto-falls back to `DAP_IO_FLOW_LB_TIER_APPLICATION` when address is loopback.  
This test uses `TEST_TRANS_SERVER_ADDR`, which is fixed to `"127.0.0.1"` by default, so on Linux/Android it no longer runs on CBPF tier.

### Risk
The test named `test_udp_multiclient_regression` stops exercising the CBPF path it was originally intended to stress/regress.  
This creates false confidence: CBPF regressions can reappear while this test still passes.

### Recommended fix
- Keep explicit CBPF execution for this regression test (or add explicit tier matrix run).
- If loopback instability is real, split scenarios:
  - fast/stable smoke on Application tier,
  - dedicated regression path on CBPF (non-loopback or isolated environment).

### Suggested checks after fix
- Verify test logs show forced tier = `CBPF` for the regression scenario.
- Reproduce expected stress conditions on CBPF path in CI or dedicated nightly job.

### Fix applied
- Linux/Android path in `s_configure_lb_tier()` was changed to force `DAP_IO_FLOW_LB_TIER_CLASSIC_BPF` unconditionally for this regression test.
- Loopback-driven fallback to `APPLICATION` was removed from this test.
- `Fix commit`: `this-commit`.

## Finding RF-2026-04-21-009
- Priority: P2
- Status: FIXED
- Location: `tests/integration/net/trans/test_trans_integration.c:928-949`, scenarios at `:90-100`

### What is wrong
Integration worker silently caps UDP/DNS payload to `256 * 1024` bytes, while scenario table defines up to `1 * 1024 * 1024` and `512 * 1024` for default runs.

### Risk
Declared scenarios are no longer actually executed for datagram transports.  
Large-payload fragmentation/retransmit/reassembly behavior is no longer covered at intended sizes, so regressions in that range can slip through.

### Recommended fix
- Do not silently override scenario size in worker logic.
- If cap is needed, expose it explicitly in scenario definition and reporting (separate “capped” scenarios).
- Prefer per-transport scenario sets so test expectations remain explicit and comparable.

### Suggested checks after fix
- Assert effective payload equals scenario payload unless scenario is explicitly marked capped.
- Run at least one UDP and one DNS scenario at >= 512 KB in CI/nightly.

### Fix applied
- Removed hardcoded datagram payload cap (`256 * 1024`) from `test_trans_worker()`.
- Effective payload size now follows scenario definition for all transports.
- `Fix commit`: `this-commit`.
