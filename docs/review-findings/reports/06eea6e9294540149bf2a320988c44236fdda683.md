## Finding RF-2026-04-21-013
- Priority: P1
- Status: FIXED
- Location: `net/stream/stream/dap_stream.c:1422-1449`

### What is wrong
Fragment reassembly switched from strict contiguous assembly to a byte-map approach, but assembly context is keyed only by `full_size`:
- buffer is reused while `buf_fragments_size_total == full_size`,
- no fragment-group discriminator is tracked (packet/sequence id is absent in fragment header),
- completion is decided only by “all byte positions filled”.

With UDP/DNS reordering, fragments from different packets that share the same `full_size` can be merged into one assembled buffer and treated as a valid packet.

### Risk
Silent payload corruption and nondeterministic channel behavior under reordering:
- mixed packet may be delivered as valid payload,
- mixed packet may occasionally pass structural checks and propagate bad data,
- replay/loss logic becomes unreliable because header bytes can come from a different packet than body bytes.

### Recommended fix
- Do not merge fragments across packet boundaries without a discriminator.
- Safer options:
  - restore strict in-order assembly (previous behavior), or
  - extend `dap_stream_fragment_pkt_t` with a fragment-group id (for example channel seq-id) and validate it for all fragments before merging.

### Suggested checks after fix
- Send two fragmented packets with equal `full_size` and intentionally interleave/reorder fragments.
- Verify assembled output is either:
  - correct per packet, or
  - dropped/reset; never mixed.

### Fix applied
- Reassembly switched to strict sequential mode in `s_stream_proc_pkt_in()`:
  - new fragment assembly starts from `mem_shift == 0`,
  - non-zero-shift fragment without active matching assembly is dropped,
  - every next fragment must match `buf_fragments_size_filled` exactly.
- Removed byte-map-based “filled anywhere” merge behavior from this path, eliminating cross-packet mixing for equal-size interleaved fragment streams.
- `Fix commit`: `this-commit`.

## Finding RF-2026-04-21-014
- Priority: P1
- Status: FIXED
- Location: `net/stream/stream/dap_stream.c:1413-1416`, copy loop at `:1438-1448`

### What is wrong
Bounds validation uses:
- `(size_t)mem_shift + (size_t)size > full_size`

On 32-bit targets `size_t` is 32-bit, so this addition can overflow.  
After overflow, invalid ranges can pass the guard and reach:
- `memcpy(a_stream->buf_fragments + mem_shift, ...)`,
- `for (i = begin; i < end; i++) buf_fragments_map[i] = ...`.

### Risk
Out-of-bounds memory write/read on 32-bit builds from malformed network fragments (remote crash/corruption vector).

### Recommended fix
Use overflow-safe bounds logic:
- first validate `mem_shift <= full_size`,
- then validate `size <= full_size - mem_shift`,
- or perform calculation in `uint64_t` and compare before any pointer arithmetic.

### Suggested checks after fix
- Add negative test with crafted fragment fields (`mem_shift` near `UINT32_MAX`).
- Run fragment parsing tests on a 32-bit target/emulation build with ASAN/UBSAN.

### Fix applied
- Replaced overflow-prone bounds guard with subtraction-safe checks:
  - `mem_shift <= full_size`,
  - `size <= full_size - mem_shift`.
- Added post-copy guard for `buf_fragments_size_filled > full_size` with cleanup.
- `Fix commit`: `this-commit`.

## Finding RF-2026-04-21-015
- Priority: P2
- Status: FIXED
- Location: `net/stream/ch/dap_stream_ch_pkt.c:356-390` (sleep at `:390`)

### What is wrong
DNS-specific fragmentation path in `dap_stream_ch_pkt_write_unsafe()` introduces blocking pacing:
- `if (l_dns_fragment_pacing && l_remaining > l_fragment_size) dap_usleep(DAP_STREAM_DNS_FRAGMENT_PACE_US);`

`dap_stream_ch_pkt_write_unsafe()` is called from stream worker callbacks (`net/stream/stream/dap_stream_worker.c:124`, `:154`), so this sleep blocks the worker thread/event loop directly.

### Risk
Under large fragmented DNS payloads, per-fragment sleep accumulates and stalls the worker for tens/hundreds of milliseconds:
- delays processing of unrelated sockets/channels on the same worker,
- increases keepalive/retransmission jitter,
- can cause timeout-like behavior and throughput collapse under concurrent load.

### Recommended fix
- Remove blocking `dap_usleep()` from the hot path.
- Implement pacing using non-blocking scheduling:
  - per-stream token bucket / send budget, or
  - deferred next-fragment dispatch via worker timer/callback.
- If temporary pacing is required, make it configurable and disabled by default in production paths.

### Suggested checks after fix
- Run concurrent stress with:
  - one stream sending large fragmented DNS packets,
  - parallel control traffic/keepalive streams on same worker.
- Verify latency and keepalive stability do not regress compared to non-fragmented baseline.

### Fix applied
- Removed DNS-specific blocking pacing from `dap_stream_ch_pkt_write_unsafe()`:
  - deleted `DAP_STREAM_DNS_FRAGMENT_PACE_US` constant,
  - deleted `l_dns_fragment_pacing` branch,
  - removed `dap_usleep(...)` from fragmentation loop.
- Fragment send path no longer performs per-fragment blocking sleep in stream worker context.
- `Fix commit`: `this-commit`.
