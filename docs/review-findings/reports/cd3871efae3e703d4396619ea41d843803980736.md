## Finding RF-2026-04-21-001
- Priority: P1
- Status: FIXED
- Location: `plugin/src/dap_plugin.c:262-276`

### What is wrong
`dap_plugin_start()` does:
1. `s_load(l_manifest)`
2. `s_preinit(l_module)`
3. `s_init(l_module)`
4. returns `0`

Return codes from `s_preinit()` and `s_init()` are ignored.

### Risk
A plugin can fail during preinit/init but still be reported as started. This can leave partially initialized runtime state and break operational control flow (CLI automation, restart logic, health expectations).

### Recommended fix
- Capture and propagate return codes from `s_preinit()` and `s_init()`.
- On failure, either:
  - unload module and return error code, or
  - keep module loaded but mark explicit failed state (requires status model extension).
- For current API, simplest safe behavior: return first non-zero error code and avoid reporting success.

### Suggested checks after fix
- plugin preinit returns error -> `dap_plugin_start()` must return error.
- plugin init returns error -> `dap_plugin_start()` must return error.
- successful path still returns `0`.

## Finding RF-2026-04-21-002
- Priority: P2
- Status: FIXED
- Location: `core/include/dap_common.h:520-531` (enum), references in `core/src/dap_common.c` and `global-db/dap_global_db.c`

### What is wrong
`L_TPS` is not present in `dap_log_level_t`, but code under `#ifdef DAP_TPS_TEST` still uses `L_TPS`.

### Risk
With `DAP_TPS_TEST` enabled, this creates compile-time breakage or inconsistent log-level contract.

### Recommended fix
Use one consistent approach:
- Option A: restore `L_TPS` in enum (guarded or always present).
- Option B: remove/replace all `L_TPS` usages and map TPS logs to another existing level.

For minimal change and backward compatibility, Option A is usually safer.

### Suggested checks after fix
- Build with `DAP_TPS_TEST=ON` and `OFF`.
- Confirm log level arrays and enum indexes remain aligned.

## Finding RF-2026-04-21-003
- Priority: P3
- Status: FIXED
- Location: `io/dap_events_socket.c:1744-1753`

### What is wrong
POSIX queue path uses:
- `ret = mq_timedsend(...)`
- success check: `ret == sizeof(a_arg)`

But POSIX `mq_timedsend()` returns `0` on success and `-1` on error.

### Risk
Successful sends are treated as failures in this branch; caller may get false error code and fallback logic may be triggered incorrectly.

### Recommended fix
- Treat `ret == 0` as success.
- Read `errno` only on `ret == -1`.
- Return normalized status consistently with other branches.

### Suggested checks after fix
- Fill queue with valid capacity and verify send returns success.
- Force timeout/full queue and verify correct non-zero error propagation.
