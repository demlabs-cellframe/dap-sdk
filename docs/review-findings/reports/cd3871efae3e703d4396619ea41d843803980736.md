## Finding RF-2026-04-21-001
- Priority: P1
- Status: FIXED
- Location: `plugin/src/dap_plugin.c:262-276`

### What is wrong
`dap_plugin_start()` does:
1. `s_load(l_manifest)`
2. `s_preinit(l_module)`
3. `s_init(l_module)`
4. returns `0`

Return codes from `s_preinit()` and `s_init()` are ignored.

### Risk
A plugin can fail during preinit/init but still be reported as started. This can leave partially initialized runtime state and break operational control flow (CLI automation, restart logic, health expectations).

### Recommended fix
- Capture and propagate return codes from `s_preinit()` and `s_init()`.
- On failure, either:
  - unload module and return error code, or
  - keep module loaded but mark explicit failed state (requires status model extension).
- For current API, simplest safe behavior: return first non-zero error code and avoid reporting success.

### Suggested checks after fix
- plugin preinit returns error -> `dap_plugin_start()` must return error.
- plugin init returns error -> `dap_plugin_start()` must return error.
- successful path still returns `0`.

## Finding RF-2026-04-21-003
- Priority: P3
- Status: FIXED
- Location: `io/dap_events_socket.c:1744-1753`

### What is wrong
POSIX queue path uses:
- `ret = mq_timedsend(...)`
- success check: `ret == sizeof(a_arg)`

But POSIX `mq_timedsend()` returns `0` on success and `-1` on error.

### Risk
Successful sends are treated as failures in this branch; caller may get false error code and fallback logic may be triggered incorrectly.

### Recommended fix
- Treat `ret == 0` as success.
- Read `errno` only on `ret == -1`.
- Return normalized status consistently with other branches.

### Suggested checks after fix
- Fill queue with valid capacity and verify send returns success.
- Force timeout/full queue and verify correct non-zero error propagation.

## Finding RF-2026-04-21-004
- Priority: P1
- Status: FIXED
- Location: `net/stream/stream/dap_stream.c:1529-1550`, cleanup block at `:1594-1598`

### What is wrong
When esocket gets detached during packet callbacks, function returns early:
- after `packet_in_callback`
- after notifier callback loop

These returns bypass shared cleanup at function end (`DAP_DEL_Z(a_stream->pkt_cache)` and fragment reset path).

### Risk
Memory/state cleanup can be skipped on teardown race path, causing leaked packet cache and stale fragment state.

### Recommended fix
- Replace early `return` with jump/flag to unified cleanup epilogue.
- Keep teardown guard behavior, but ensure cleanup always executes.

### Fix applied
- Replaced early `return` branches with `goto cleanup;` so shared cleanup path always executes.
- `Fix commit`: `this-commit`.

### Suggested checks after fix
- Simulate detach inside channel callback and verify `pkt_cache` is cleared.
- Simulate detach inside notifier callback and verify fragment state is reset when required.

## Finding RF-2026-04-21-006
- Priority: P2
- Status: FIXED
- Location: `crypto/src/dap_sign.c:583-585`

### What is wrong
`dap_sign_get_information()` uses inverted condition for output format selection:
- for `a_hash_out_type == "hex"` it prints Base58
- for non-hex it prints hex

`dap_sign_get_information_json()` uses the opposite (correct) logic, so text and JSON outputs diverge.

### Risk
Operators and automation may request hex output and receive Base58 in text mode, causing parsing errors, false comparisons, and inconsistent diagnostics between CLI and JSON interfaces.

### Recommended fix
- Align text output logic with JSON output logic:
  - `hex` -> `dap_chain_hash_fast_to_str_static()`
  - non-hex -> `dap_enc_base58_encode_hash_to_str_static()`

### Fix applied
- Replaced inverted condition in `dap_sign_get_information()` with the same condition pattern already used in JSON variant.
- `Fix commit`: `this-commit`.

### Suggested checks after fix
- Call with `a_hash_out_type = "hex"` and verify text output is hex.
- Call with non-hex output type and verify text output is Base58.
- Compare text and JSON outputs for the same signature and output type.

## Finding RF-2026-04-21-007

- Location: `global-db/dap_global_db_cluster.c:64-73`

### What is wrong
`dap_global_db_cluster_deinit()` was changed to a no-op (only static pointer reset), removing list iteration and `dap_global_db_cluster_delete()` calls.

### Risk

### Recommended fix
- Restore explicit cluster cleanup in `dap_global_db_cluster_deinit()`:
  - iterate `l_dbi->clusters` with `DL_FOREACH_SAFE`
  - call `dap_global_db_cluster_delete()` for each item
  - reset static pointers after cleanup

### Fix applied
- Reintroduced cluster iteration and deletion in `dap_global_db_cluster_deinit()`.
- Kept `s_local_cluster`/`s_global_cluster` reset after cleanup.
- `Fix commit`: `this-commit`.

### Suggested checks after fix

- Validate no remaining clusters in instance list after deinit.
- Run with ASAN/LSAN to confirm cleanup path.
