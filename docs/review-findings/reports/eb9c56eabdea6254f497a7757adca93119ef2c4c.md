- Priority: P2

- Location: `net/trans/dns/dap_net_trans_dns_server.c:317-337`, call site `net/trans/dap_net_trans_server.c:219-238`

### What is wrong
`dap_net_trans_dns_server_delete()` now aborts and returns early when:
- `s_dns_server_stop_internal()` reports incomplete stop, or
- `pthread_rwlock_destroy()` fails.

At the same time, generic wrapper deletion in `dap_net_trans_server_delete()` always frees the wrapper object after invoking `ops->delete()` and does not receive any status from the transport-specific delete callback.

As a result, on these early-return branches the DNS-specific object is left allocated, but its owner wrapper is already freed, so the pointer to retry cleanup is lost.

### Risk
- Leaked DNS server object/resources on shutdown under load/race conditions (for example when stop reports in-flight reads not drained in time).
- Potential accumulation across repeated start/stop cycles.
- Cleanup semantics become inconsistent with other transport delete paths that guarantee object release.

### Recommended fix
- Prefer making transport-specific `delete` callbacks non-failing from caller perspective: always finalize/free object in `dap_net_trans_dns_server_delete()`, using best-effort stop and explicit forced cleanup fallback.
- If failure reporting is required, extend server-ops API to return status and let wrapper decide whether it is safe to free outer object.

### Suggested checks after fix
- Force `s_dns_server_stop_internal()` into incomplete-stop path (artificially delayed read processing), then run delete flow and verify no leaked DNS server allocations.
- Run repeated DNS trans server create/start/stop/delete cycles under ASAN/LSAN.

### Fix applied
- `dap_net_trans_dns_server_delete()` no longer aborts on partial stop failure.
- Delete path now:
  - forces `stopping=true`,
  - ensures listener `dap_server_delete_sync()` is executed if still present,
  - waits in bounded rounds until `datagram_reads_inflight == 0`,
  - performs session cleanup in delete fallback path,
  - proceeds with best-effort lock destroy and always releases DNS server object.
- `Fix commit`: `this-commit`.
